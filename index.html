🎯 What is E-Crime Hunting & Disruption?

E-Crime Hunting & Disruption is a multidisciplinary investigative framework that extends beyond conventional incident response. It integrates analysis of technical artifacts with financial tracing to generate actionable intelligence aimed at identifying, tracking, and disrupting adversaries and their supporting infrastructure.

Rather than a fixed role, it represents a convergence of capabilities: malware reverse engineering, enterprise cloud forensics, incident response, threat hunting, and blockchain-based financial analysis. These disciplines are applied collectively to correlate attacker activity across both the public internet and underground ecosystems.

The objective is not only containment, but also attribution, infrastructure disruption, and actionable threat intelligence — ensuring that criminal operations incur both operational and legal consequences.

Per umbras numerosque venari
"To hunt through shadows and numbers" — the investigator's mandate.

Convergence of Disciplines

🔬 Technical Forensics
Malware analysis, reverse engineering, cloud artifact collection, disk/memory forensics, and network packet analysis to reconstruct attacker TTPs.
⛓️ Blockchain Analysis
On-chain transaction tracing, wallet clustering, exposure analysis, and VASP attribution to follow illicit financial flows across crypto ecosystems.
🕵️ OSINT Intelligence
Open-source intelligence gathering from social media, breach databases, WHOIS records, and underground forums to correlate on-chain data with real-world identities.
⚖️ Legal Disruption
Translating forensic evidence into legal action — subpoenas, asset freezes, seizure orders, and international law enforcement cooperation.
🔐 Threat Hunting
Proactive discovery of adversary infrastructure, indicator pivoting, and network mapping to expose criminal operations before victims are targeted.
📊 Financial Intelligence
Anti-money laundering (AML) pattern recognition, PCMLTF analysis, and financial crime typology mapping to trace proceeds through complex laundering chains.

🔑 Core Investigator Principles
  • Follow the money: The blockchain never lies. Every transaction is permanently recorded. The investigator's job is to read the ledger.
  • Pseudonymous ≠ Anonymous: Wallet addresses don't contain names, but behaviors, patterns, and mistakes leave an indelible trail.
  • On-chain + off-chain: The strongest investigations combine blockchain data with OSINT, social media, breach data, and traditional intelligence.
  • Speed matters: Funds move in minutes. The faster the trace, the more likely an exchange can freeze assets before they're cashed out.
  • Disruption is the goal: Attribution alone is not enough. Every investigation should aim to impose operational and legal consequences.
  • Documentation is evidence: Screenshot everything with timestamps. Archive web pages. Every note could become a court exhibit.

Blockchain Foundations

You cannot investigate what you don't understand. This module builds foundational knowledge of how blockchains work and why they are investigable — despite the perception of anonymity.

⛓️ What is a Blockchain?

A blockchain is a publicly available, immutable ledger of every transaction ever made on that network. Think of it as a global spreadsheet that anyone can read, but from which no one can delete rows.

When Person A sends Bitcoin to Person B, that transaction is cryptographically verified by thousands of computers worldwide and permanently written into a "block" — chained to every previous block. This transparency is what makes blockchain investigation possible. Unlike traditional financial crime where investigators need warrants to access bank records, on-chain data is publicly visible to everyone.

How Transactions Work

STEP 01
Wallet Creation
A user creates a crypto wallet, generating a private key (a secret password) and a public address (like a bank account number). The address is pseudonymous — it doesn't contain a name, but it is permanently traceable.
STEP 02
Transaction Broadcast
When sending crypto, the sender broadcasts a signed transaction to the network: "I'm sending 0.5 BTC from address 1ABC… to address 3XYZ…." This is signed with the private key, proving ownership of funds without revealing the key itself.
STEP 03
Network Verification
Network nodes verify the transaction is valid — the sender has sufficient funds and the signature is cryptographically correct. Valid transactions enter the mempool (pending queue).
STEP 04
Block Confirmation
Miners or validators bundle pending transactions into a block and add it to the chain. Bitcoin: ~10 min. Ethereum: ~12 sec. Once confirmed, the transaction is permanent and irreversible.
STEP 05
Public Permanent Record
The transaction is now visible to anyone using a block explorer: sender address, recipient address, amount, timestamp, and transaction hash (TXID). This is the investigator's primary evidence source.

🔎 Why Blockchain IS Traceable — The Investigator's Advantage

Criminals often believe crypto is "anonymous." This misconception is their vulnerability and the investigator's advantage:

  • Every transaction is public — unlike cash, there is a permanent, immutable audit trail for every movement
  • Addresses cluster together — one person controls multiple addresses that exhibit identifiable behavioral patterns
  • Exchanges require KYC — to convert crypto to fiat, criminals must eventually pass through identity verification
  • Metadata leaks identity — IP addresses, device IDs, timing patterns, and transaction errors expose real-world presence
  • Criminals brag — social media posts, luxury spending, and forum activity create OSINT leads that link pseudonyms to real people

Key Terminology

Master the language of blockchain investigation. Sourced from TRM Labs' Blockchain Investigator's Flip Book and the FACT Attribution Framework — the field standards used by law enforcement worldwide.

📚 Sources
Core definitions sourced from the TRM Labs Blockchain Investigator's Flip Book and FACT Attribution Framework v1.1. All terms also appear in the searchable Glossary (Reference section).
Attribution

The process of labeling or assigning specific blockchain addresses to specific real-world entities. For example: identifying that address bc12345 belongs to Binance. Attribution is the primary goal of blockchain investigation. Sources include: exchange records, court filings, law enforcement databases, commercial analytics platforms, and OSINT.

Blockchain Tracing

Tracking the source and destination of cryptocurrency transactions recorded on a blockchain. The investigative goal is to identify the actual controller of an otherwise pseudo-anonymous address by following money to or from entities that hold real-world identity data (VASPs).

Chain-Hopping

Moving assets between different blockchains or cryptocurrencies many times to obscure asset control. Example: sending illicit Bitcoin through a cross-chain bridge in exchange for "clean" Ether on Ethereum, then bridging again to Tron. Investigators must follow the trail across multiple chains and tools.

Counterparty

The party on the opposite side of a transaction. If "A" sends Bitcoin to "B," then B is A's counterparty. Identifying counterparties — especially known VASPs or labeled entities — is a core tracing strategy. The counterparty is often the leverage point for legal process.

Exposure (Direct vs. Indirect)

Direct exposure: Address A sent directly to Address B (one hop). Indirect exposure: A→B→C, where C has indirect exposure to A (two or more hops). Commercial tools quantify exposure as a percentage. A high exposure score to a sanctioned entity or mixer is a significant red flag.

Transaction Hash (TXID)

A unique alphanumeric identifier for a specific blockchain transaction — the receipt number. Essential for looking up any transaction on a block explorer. Example BTC TXID: 4a5e1e4baab89f3a32518a88c31bc87f618f76673e2cc77ab2127b7afdeda33b

VASP — Virtual Asset Service Provider

Any platform used to buy, sell, trade, or exchange virtual currency — exchanges like Binance, Coinbase, Kraken. Most VASPs collect KYC data that investigators can obtain via legal process. The FATF term, used interchangeably with "exchange" in most jurisdictions.

KYC — Know Your Customer

Real-world identification collected by VASPs: government ID, selfie, date of birth, address, phone, email. When a VASP is served legal process and provides KYC records, it typically identifies the real person controlling an otherwise anonymous wallet — the breakthrough that cracks most cases.

Mixers & Tumblers

Services that obfuscate fund origins by pooling funds from multiple users and redistributing them — breaking the linear chain of association. Examples: Tornado Cash (ETH, OFAC-sanctioned 2022), Wasabi Wallet (BTC, CoinJoin). Using a mixer does not make funds untraceable; it creates a fork in the investigation requiring parallel analysis of all outputs.

Block Explorer

A publicly accessible tool for querying and visualizing blockchain data — transaction history, address balances, relationships, and timestamps. Examples: Etherscan (ETH), Blockchair (BTC), Solscan (SOL). Commercial tools like TRM Labs and Arkham add entity labels and risk scores on top of raw data.

Pass-Through Address

An address that receives funds and immediately forwards the entire amount, leaving a $0 balance. A significant red flag — the controller paid transaction fees for an economically unnecessary step, suggesting deliberate layering. Key indicators: only two transactions (in + out), large amounts, rapid succession (minutes apart).

Address Clustering

Grouping multiple blockchain addresses likely controlled by the same entity using heuristics. Primary method: Common Input Ownership (CIUA) — if two addresses both sign inputs in the same Bitcoin transaction, the same private key holder controls both. Change address detection and spending pattern analysis extend the cluster further.

🚧 How Criminals Attempt to Hide Funds

Peeling Chains

Sending funds through many sequential addresses, each time peeling off a slightly smaller amount to confuse investigators. Creates a long chain of linked addresses that must each be analyzed.

Layering

Moving funds through multiple transactions, services, and blockchains to create maximum distance between illicit source and final destination. The "L" in the AML Placement → Layering → Integration cycle.

Instant / No-KYC Exchanges

Services like ChangeNow and FixedFloat allow swapping cryptocurrencies without identity verification. No central operator to subpoena, but all on-chain transactions remain traceable.

DeFi Protocol Routing

Running funds through decentralized exchanges (DEXs) or lending protocols. No central custodian to serve legal process to — but all activity is on-chain, timestamped, and publicly traceable.

Blockchain Tracing in Practice

The step-by-step methodology for tracing illicit funds — from acquiring an address to identifying a real person. Based on TRM Labs' official investigator methodology.

🗺️ The Investigative Goal

The goal of blockchain tracing is to identify the actual controller of an otherwise pseudo-anonymous address. To do so, trace blockchain transactions to find counterparty exposure with an entity or individual that can provide the identity of the beneficial controller. More simply: follow the money to or from an entity that can provide real-world identity.

The Three-Step Framework

1
Identify Illicit Conduct

Acquire the address/transaction from SAR, complaint, OSINT, or source. Confirm illicit activity is present and document the evidence trail from the outset.

2
Trace Proceeds

Follow the flow of funds through the blockchain, mapping the transaction network, identifying leverage points and third-party custodians with KYC data.

3
Seek Disruption

Serve legal process on VASPs to obtain KYC data. Seek emergency asset freezes, seizure orders, and ultimately arrest or civil forfeiture.

Detailed Tracing Steps

A
Input Transaction into Blockchain Tool
Copy/paste the transaction hash, addresses, or other information into a blockchain tracing tool (TRM Labs, Arkham, Etherscan). Best practice: acquire the address electronically to avoid transcription errors. Many tools auto-populate even partial addresses.
B
Identify Leverage Points — Map the Network
Ask: How many transactions? When were the most recent? How much value is currently held? Is it part of a larger cluster? Does it have attribution or risk flags? Does it have exposure to a VASP that keeps KYC records?
C
Identify Third Parties for Legal Process
After plotting the address and its direct counterparties, identify which VASPs or custodians have exposure to the illicit funds. Screenshot the graph as evidence explaining to a prosecutor why legal process should be served on a specific exchange.
D
Serve Legal Process & Seek Freeze
Issue subpoenas or court orders to VASPs requesting KYC documents, subscriber info, email associations, IP address logs, full transaction history, device IDs, and associated accounts. Request simultaneous asset freeze where funds remain on the platform.
🚩 Red Flag: Pass-Through Addresses

A pass-through address is a significant indicator of illicit layering. Key patterns: only two transactions (one in, one out of the entire sum received), large amounts moved in rapid succession (e.g., 39 minutes apart), $0 current balance, part of a large cluster. This represents a non-economic trade — the controller paid fees for an unnecessary step.

Example from TRM's Flip Book: bc1q5spf8v3lwxcnxy9ep0mcxppy3qzdynk53vs9k7 — linked to a romance fraud scheme that received and immediately sent $154,814.

OSINT Techniques

Open Source Intelligence bridges the gap between a pseudonymous wallet address and a real-world identity. These are the methods used by ZachXBT and professional investigators.

"ZachXBT's investigative approach combines blockchain forensics with OSINT techniques... cross-referencing on-chain data with public records such as domain registrations, court filings, and social media activity. He also monitors online forums and chat groups on Telegram and Discord where cybercriminals congregate." — Wikipedia profile of ZachXBT (March 2026)

The OSINT Investigation Workflow

PHASE 1
On-Chain Pivot Points
Start with what the blockchain provides: wallet addresses, transaction hashes, amounts, timing. Look for exchange deposits (requiring KYC), ENS/domain names, NFT mints, or labeled addresses. These are the first pivot points into the real world.
PHASE 2
Social Media Intelligence
Criminals brag. Search Twitter/X, Telegram, Discord, and Reddit for mentions of wallet addresses, amounts, project names, or usernames. ZachXBT identified $243M BTC theft suspects partly through their social media spending — luxury purchases and nightclub appearances inconsistent with any legitimate income.
PHASE 3
Username & Email Pivots
A username used in one context often reappears elsewhere. Use OSINT Industries, Sherlock, and IntelX to find all accounts associated with a username, email, or phone number. Breach databases (Snusbase, LeakPeek) may reveal passwords, IPs, or real names — often the break that cracks a case.
PHASE 4
Domain & Infrastructure Intel
WHOIS records, SSL certificate history, Shodan/Censys scans, and domain registration data can link a scam website to a real identity or to other criminal infrastructure. The Lazarus Group registered bybit-assessment[.]com using trevorgreer9312@gmail.com hours before the $1.5B Bybit hack — a traceable footprint.
PHASE 5
Physical Evidence Correlation
Instagram story locations, EXIF data embedded in photos, check-ins, flight patterns, and purchases can place a suspect in a physical location. ZachXBT confirmed a suspect's location in Abbotsford, BC, through Instagram story posts. OSINT on Khinkis revealed travel patterns to Southeast Asia and Australia.
PHASE 6
Archive & Preserve Evidence
Use Wayback Machine and Archive.today to preserve evidence before suspects delete it. Screenshot with timestamps. Organize everything in Obsidian or a similar link-analysis tool. Archive web pages immediately upon discovery — evidence disappears fast once suspects detect investigation.
⏱️ Timing Analysis

Matching exact blockchain transaction timestamps to social media posts, login events, or real-world activity. ZachXBT used timing analysis to match Bitcoin transfers to Tron wallet addresses in the Khinkis ransomware investigation by cross-referencing blockchain timestamps against exchange withdrawal times.

Key timing red flags: funds moving within minutes of a hack announcement; transactions at unusual hours indicating specific timezone; rapid cascading transactions suggesting automated laundering scripts; dormant addresses suddenly activating after months (criminal cashing out).

🕸️ Address Clustering Heuristics
  • Common Input Ownership (CIUA): If two addresses both sign inputs in the same Bitcoin transaction, the same private key holder controls both — they belong to the same cluster.
  • Change Address Detection: Bitcoin transactions often send "change" back to the sender. Identifying change addresses reveals more wallets controlled by the same entity.
  • Behavioral Pattern Matching: Addresses that receive from the same counterparties, or send funds together repeatedly, often belong to the same cluster even without CIUA evidence.

Economic Crimes

Financially motivated crimes that exploit cryptocurrency's pseudonymity, global reach, and irreversibility. Each category includes definition, crypto-crime nexus, key indicators, and investigation pointers.

The Fraud Diamond

Fraud Diamond — Four Elements Required for Fraud to Occur
Pressure / Incentive
The motivation driving the fraudster — financial desperation, debt, lifestyle pressure, or greed. In crypto fraud: the promise of "easy money" or the need to cover losses from previous schemes.
Opportunity
The circumstances enabling fraud — weak controls, anonymous systems, or exploitable trust. Crypto's pseudonymity, global reach, and irreversible transactions create abundant opportunity for fraudsters.
Rationalization
How the fraudster justifies the act to themselves — "victims are greedy," "it's a victimless crime," or "the system is rigged." Crypto fraudsters often rationalize by blaming victim FOMO or greed.
Capability
The fourth element (added to the original Fraud Triangle) — the skill, knowledge, and access required to execute the fraud. In crypto: technical knowledge of wallets, smart contracts, or social engineering skills.

Romance scams / Pig Butchering (Sha Zhu Pan — "killing the pig") are long-con social engineering frauds originating predominantly from Southeast Asian cyber-scam compounds. Criminals build a romantic or investment friendship online over weeks or months, cultivate deep trust, then convince victims to "invest" in a fraudulent crypto trading platform they control.

Business Email Compromise (BEC) is a sophisticated fraud targeting organizations that conduct wire transfers or make business payments. Attackers compromise or spoof legitimate business email accounts — typically executives (CEO fraud), vendors, or legal counsel — and impersonate them to redirect payments to criminal-controlled accounts.

BEC has evolved to increasingly request cryptocurrency payments directly, or use compromised accounts to change payment instructions from bank wire to crypto wallet. The FBI's IC3 reports BEC as consistently among the highest-loss cybercrime categories globally.

Crypto Redirect Method
Criminal intercepts legitimate vendor invoice, replaces bank account with crypto wallet address. Victim sends payment to criminal wallet believing it is vendor.
Why Crypto
Irreversibility; speed (minutes vs. days for wire recalls); no central authority to contact for reversal; easier international transfer than SWIFT

  • Look-alike / typosquat domain spoofing
  • Infostealer malware credential theft
  • VoIP spoofing to accompany fraudulent email

Accounts payable staff, CFOs, law firm trust accounts, real estate transaction accounts, and any high-value payment workflow without multi-factor verification

  • NFT self-purchase to move value (wash trading)

Money mules are individuals who receive and transfer illicit proceeds — knowingly or unwittingly — on behalf of criminal networks. In crypto, mule networks provide an additional identity layer between the criminal and the VASP's KYC records.

Definition

Ransomware is malicious software that encrypts a victim's files or systems, rendering them inaccessible, and demands payment (typically in cryptocurrency) in exchange for the decryption key. Modern ransomware-as-a-service (RaaS) operations involve multiple criminal parties: the RaaS developer, affiliates who deploy the malware, and brokers who launder proceeds.

Triple extortion adds DDoS attacks. Nation-state actors (Lazarus Group, Russian RaaS affiliates) use ransomware to fund operations or generate plausible deniability.

Payment Architecture
  • Unique Bitcoin address generated per victim
  • Monero (XMR) preferred for anonymity in some groups
  • Payment portal accessible via TOR hidden service
  • Automatic decryption key delivery upon payment confirmation
Laundering Flow
  • Victim pays → per-victim BTC address
  • Consolidation to group wallet
  • Mixing/chain-hopping cycle
  • OTC broker fiat conversion (documented in Khinkis case)
Why Crypto
Irreversible; censorship-resistant; global; automated payment verification; allows RaaS profit-sharing between developer (20-30%) and affiliates (70-80%) via smart splits
Major Groups
LockBit, ALPHV/BlackCat, Cl0p, REvil, Conti (disbanded), Royal, Lazarus Group (DPRK-state), WannaCry (DPRK). Most operate RaaS models with affiliate programs.
Technical IOCs
  • Encrypted files with unusual extensions (.locky, .ryuk, .crypt)
  • Ransom note dropped in directories
  • Shadow copies deleted (vssadmin delete shadows)
  • RDP brute force in logs preceding infection
  • Cobalt Strike beacon activity pre-encryption
Network IOCs
  • C2 communication to TOR exit nodes
  • Large data exfiltration prior to encryption
  • Lateral movement using PSExec or WMI
  • Kerberoasting or credential dumping activity
Financial IOCs
  • Bitcoin payment demanded within 72 hours
  • Ransom amount calibrated to victim size (negotiated)
  • Payment address with 15%+ exposure to other ransomware victims
Attribution Indicators
  • Ransom note language/format matches known group
  • TTPs match known group playbook (MITRE ATT&CK)
  • Bitcoin payment address cluster matches prior incidents

Compare ransom note format, payment address behavior, and TTPs against known ransomware group signatures. TRM Labs and Chainalysis maintain ransomware group address databases for rapid attribution.

Sequential transaction cascade where each address forwards to the next with slightly less value — creating dozens of hops to exhaust investigative resources. Commercial tools (TRM Labs) trace these automatically; manual analysis requires systematic address-by-address documentation.
Instant / No-KYC Exchanges
ChangeNow, FixedFloat, SimpleSwap — swap services with no identity verification. No operator to subpoena. However, these services typically have consistent wallet infrastructure that can be identified and tracked across multiple investigations.
NFT Wash Trading
Buying NFTs between self-controlled wallets at inflated prices to create transaction records establishing "legitimate" provenance for funds. The NFT then appears to have earned its purchase price through the art market.
VPN / TOR
Obscures real IP address during VASP logins and blockchain interactions. Spur.us identifies VPN/TOR IPs in VASP subpoena returns. Residential proxy services are harder to detect but leave billing trails.
Non-Custodial Wallets
MetaMask, Phantom, Ledger — wallets where the user holds private keys. No central custodian to subpoena. However, the wallet address itself is on-chain and investigable. The challenge is linking the address to a real identity.
Privacy Coins at Bridges
Converting to Monero (XMR) at a bridge point creates the hardest forensic break. The Monero transaction is opaque — but the bridge-in and bridge-out transactions remain on the public blockchains on either side, creating investigable chokepoints.
Multi-Sig Wallets
Requiring multiple private keys to authorize a transaction — distributing control across multiple jurisdictions or individuals to complicate seizure. The Bybit hack exploited a compromised multi-sig infrastructure rather than bypassing it.
1
Systematic Hop Documentation
Document every address in the layering chain with timestamps, amounts, and counterparties. Each hop is an evidence exhibit. Use TRM Labs graphs for automatic multi-hop visualization. Screenshot and archive at each step.
2
Bridge Tracking Tools
Use Pulsy, Socketscan, and chain-specific bridge explorers to follow cross-chain transfers. Identify source and destination of every bridge transaction — the bridge-out address is investigable on the destination chain.
3
Mixer Analysis
When funds enter a mixer, use output timing analysis and amount correlation (funds going in vs. amounts coming out in similar denominations) to identify likely output addresses. Tornado Cash withdrawal patterns have been analyzed to deanonymize outputs in multiple cases.
4
Privacy Coin Chokepoints
Even if XMR internal transactions are opaque, trace what entered and exited the XMR ecosystem. Exchange deposits of XMR require KYC at compliant VASPs. Find the exchange where XMR was purchased or cashed out and serve legal process.

Legal Process & VASP Cooperation

Translating blockchain evidence into legal action — subpoenas, court orders, international cooperation, and working with VASPs. Based on TRM Labs' Flip Book and DOJ guidelines.

⚖️ What You Can Request From a VASP

When issuing legal process (subpoena, production order, MLAT request) to a VASP, request all of the following:

  • KYC documents — government ID, selfies, address
  • Subscriber information — name, DOB, registration date
  • Email address associations
  • IP address logs — login history with timestamps
  • Full transaction history — all deposits/withdrawals
  • Device ID (IMEI) — phone/device fingerprints
  • Account balance — current holdings (critical for freeze)
  • Associated accounts — linked family/business accounts

Best practice: Request an omnibus "all records associated with the account and registrant" to capture anything not specifically anticipated. Simultaneously request an emergency asset freeze if funds remain on platform.

Downstream Investigative Actions

1
Email Search Warrants
VASPs send transaction confirmation emails. Serve a search warrant to the email provider (Google, Microsoft) for email content — independently corroborating that the account holder was aware of and participated in transactions.
2
IP Address Intelligence
Cross-reference VASP-returned IP addresses with Spur.us to determine if they are residential, VPN, or TOR. Residential IPs lead to ISP subpoenas for subscriber identity — name and civic address of the account holder.
3
Asset Freeze Requests
If funds remain on the VASP, request an emergency freeze simultaneous with the data request. Speed is critical — funds may be withdrawn during the compliance window. Some VASPs maintain direct law enforcement hotlines for emergency freezes.
4
International Cooperation (MLATs)
For foreign-jurisdiction VASPs, use Mutual Legal Assistance Treaties (MLATs) or direct law enforcement channels (FBI, Europol, INTERPOL). Many VASPs will also respond to well-documented informal requests for emergency freezes even before formal process is served.
🚨 Straw Identities Warning
The named KYC account holder could be: (1) the actual subject, (2) a straw identity (friend/family), (3) a stolen identity, or (4) a professional money mule. The KYC return is a lead, not a conclusion — always investigate further.

Asset Seizure & Forfeiture

The legal and practical mechanics of seizing, holding, and forfeiting cryptocurrency — from TRM Labs' Flip Book and DOJ Asset Forfeiture Program guidelines.

🔐 Why Seize Crypto Assets?
  • Punish criminal behavior and remove ill-gotten gains from criminal control
  • Return assets to victims of the underlying crime
  • Deter future illegal activity by removing the financial profit motive
  • Disrupt criminal organizations' financial infrastructure and operating capital
  • Generate public deterrence through visible consequences for crypto crime

Custody Options for Seized Assets

Hardware Wallet

Pros: Industry standard; more secure than software wallets; portable; relatively low cost.

Cons: May not support all currencies (especially thinly-traded assets); single point of failure if device lost.

Third-Party Custodian

Pros: Expert custody, institutional-grade security, insurance options, multi-signature controls.

Cons: Transfer complexity; custodian fees; requires due diligence on custodian's own security posture.

VASP Custody

Pros: Efficient when seizing from that VASP; market-rate liquidation access.

Cons: Assets may be commingled; VASPs have been hacked; requires confidentiality safeguards to prevent subject withdrawal.

Online Software Wallet

Pros: Quick deployment; wide asset compatibility including thinly-traded currencies.

Cons: Higher intrusion risk; not recommended for large seizures; operational security burden on agency.

Five Pre-Seizure Considerations

A
Custody Type
What wallet/custodian will the agency use? Match security level to asset value. Large seizures (>$1M) warrant third-party institutional custody. Hardware wallets appropriate for mid-range. Never use software wallets for high-value seizures.
B
Access Control
Who will have access? Implement need-to-know controls. Log every access event. Avoid single point of failure — if one officer leaves, the agency retains access. Multi-signature custody recommended for large holdings.
C
Asset Form
Accept crypto as-is or convert to fiat equivalent? Consider price volatility. Thinly-traded currencies may be illiquid. Prosecutors and courts require guidance on value at time of seizure vs. time of forfeiture.
D
Contingencies
Virtual currencies are technically nascent — software becomes obsolete, VASPs get hacked, blockchains fork, protocols upgrade. Maintain a living policy document for seizure procedures. Review annually.
E
Liquidation Plan
VASP market-rate sale, third-party auction (GSA), or victim restitution in crypto? Each has tradeoffs. Returning crypto to fraud victims unfamiliar with wallets requires careful planning. Some jurisdictions require court approval before liquidation.

Glossary

🔍
Account Takeover (ATO)
Unauthorized access to and control of another person's online account — typically via stolen credentials, phishing, or SIM swap. In crypto contexts, ATO of exchange accounts leads to immediate unauthorized withdrawals.
Address Clustering
Grouping multiple blockchain addresses likely controlled by the same entity using heuristics such as Common Input Ownership (CIUA) and change address detection. A core technique for expanding investigation scope.
APT (Advanced Persistent Threat)
A sophisticated, typically state-sponsored threat actor conducting long-term targeted attacks. Lazarus Group (DPRK) is the most prominent APT in crypto crime, having stolen $2B+ in 2025 alone.
ATO
See Account Takeover.
Attribution
Assigning a blockchain address to a specific real-world entity or individual. The primary goal of blockchain investigation. Sources include: exchange KYC records, court filings, commercial analytics platforms, and OSINT.
BEC (Business Email Compromise)
Fraud targeting organizations via compromised or spoofed business email accounts to redirect payments — increasingly to cryptocurrency wallets. Consistently among the highest-loss categories in FBI IC3 annual reports.
Bitcoin ATM
A physical kiosk allowing cash-to-cryptocurrency conversion. A common money laundering placement method. ATM operators are VASPs subject to AML/KYC regulations; their records are subpoenable.
Block Explorer
A tool for querying blockchain data — transaction history, address balances, and relationships. Examples: Etherscan (ETH), Blockchair (BTC), Solscan (SOL). The foundation of every on-chain investigation.
Blockchain
A publicly available, immutable distributed ledger of all completed transactions between addresses. The transparency that makes blockchain investigation possible — unlike cash, every movement is permanently recorded.
Blockchain Tracing
Tracking the source and destination of cryptocurrency transactions to identify the real-world controller of a pseudo-anonymous address. The primary methodology of crypto financial investigation.
Chain-Hopping
Moving assets between different blockchains (e.g., BTC→ETH→TRON) through cross-chain bridges to obscure asset control and complicate investigation. A primary layering technique documented in Lazarus Group post-hack operations.
Change Address
In Bitcoin's UTXO model, the output returning unspent funds back to the sender in a transaction. Identifying change addresses reveals additional wallet addresses controlled by the same entity — extending the investigation cluster.
CIUA (Common Input Ownership)
Address clustering heuristic: if two addresses both sign inputs in the same Bitcoin transaction, the same private key holder controls both. The most reliable Bitcoin clustering method.
Clipboard Hijacker
Malware that monitors the clipboard for crypto wallet address formats and replaces copied addresses with the attacker's address. Victims unknowingly send funds to the attacker.
Cold Wallet
A cryptocurrency wallet stored offline, disconnected from the internet. Considered more secure, but Bybit's $1.5B hack demonstrated that supply chain attacks on signing infrastructure can compromise cold wallet security without accessing the device directly.
Counterparty
The party on the opposite side of a transaction. If A sends to B, B is A's counterparty. Identifying counterparties — especially known VASPs — is the core strategy for obtaining KYC data via legal process.
Cryptojacking
Unauthorized use of a victim's computing resources to mine cryptocurrency for the attacker. Monero (XMR) is the most common target. Victims experience degraded performance and elevated cloud/energy costs. Mining pool records provide investigable chokepoints.
DEX (Decentralized Exchange)
A cryptocurrency exchange operating without a central authority or KYC requirements — Uniswap, PancakeSwap, Curve. No central operator to subpoena, but all activity is on-chain and traceable. Commonly used in layering phases.
Deepfake Fraud
AI-generated synthetic video, audio, or images used to impersonate real individuals for financial deception. Applications include KYC bypass at exchanges, fake investment endorsements, real-time CEO impersonation fraud, and romance scam personas.
DeFi (Decentralized Finance)
Financial services built on blockchain smart contracts without central operators. Increasingly used for laundering due to lack of KYC — but all activity is on-chain and traceable. Smart contract exploits are a distinct attack surface.
Direct Exposure
When an address directly transacted with an illicit source — one hop away. Distinct from indirect exposure (two or more hops). Commercial tools quantify exposure as a percentage. Used in legal process justification.
Double Extortion
Ransomware tactic combining file encryption with data exfiltration — threatening to publish stolen data if ransom is not paid. Creates two leverage points. Has become standard practice among major ransomware groups since 2020.
EXIF Data
Metadata embedded in photo files including GPS coordinates, device model, and timestamps. Can reveal a suspect's physical location when photos are shared without stripping metadata. Used in multiple ZachXBT investigations.
Exposure
A quantified measure of how directly proceeds have traveled between two addresses. Commercial tools express this as a percentage. 15%+ exposure to ransomware or sanctioned entities is considered a significant investigative indicator.
FATF
Financial Action Task Force. International AML/CFT standard-setting body. Coined the term "VASP" and established the Travel Rule requiring VASPs to transmit originator/beneficiary information on crypto transactions above thresholds.
FINTRAC
Financial Transactions and Reports Analysis Centre of Canada. Canada's financial intelligence unit. Receives Suspicious Transaction Reports (STRs) from VASPs and financial institutions. Equivalent to FinCEN (US) and AUSTRAC (Australia).
Flash Loan Attack
A DeFi exploit using uncollateralized instant loans to manipulate oracle prices and drain protocols in a single atomic transaction. Entirely on-chain and traceable from the first transaction. The exploit contract deployer wallet is always investigable.
Fraud Diamond
An extension of the Fraud Triangle adding "Capability" as a fourth element. The four elements required for fraud: Pressure/Incentive, Opportunity, Rationalization, and Capability. All four must be present for fraud to occur.
Fraud Triangle
The original three-element model of fraud causation by criminologist Donald Cressey: Pressure, Opportunity, and Rationalization. The Fraud Diamond adds Capability as a fourth element.
Hawala
An informal value transfer system used predominantly in South Asian and Middle Eastern countries. Value transferred via a network of brokers (hawaladars) with settlement later. Increasingly combined with crypto to create hawala-crypto hybrid laundering systems.
Honeypot Contract
A malicious token smart contract that allows purchases but disables sells — trapping investor funds. A common rug pull variant on DEX platforms. Detectable via smart contract review tools (HoneyPot.is).
Infostealer
Malware designed to silently harvest saved credentials, browser cookies, crypto wallet files, and session tokens. Major families: Redline, Vidar, Racoon, Lumma. Sold on darknet forums via Malware-as-a-Service (MaaS) subscriptions. Logs resold on Telegram markets.
Integration
The third and final stage of money laundering — re-entering laundered funds into the legitimate economy so they appear to have a lawful origin. Crypto integration methods: OTC broker fiat conversion, real estate, luxury assets, and business investment.
Instant Exchange
A service allowing cryptocurrency swaps without KYC — ChangeNow, FixedFloat, SimpleSwap. No central operator to subpoena, but consistent wallet infrastructure is identifiable across investigations. Commonly used in layering.
KYC (Know Your Customer)
Real-world identification collected by VASPs: government ID, selfie, date of birth, address, phone, email. When obtained via legal process, KYC records typically identify the real person controlling an otherwise anonymous wallet — the most common investigative breakthrough.
Keylogger
Malware recording every keystroke — capturing passwords, seed phrases, and private keys as typed. Often bundled within infostealer packages. Seed phrase capture leads to complete and irreversible wallet compromise.
Layering
The second stage of money laundering — creating distance between funds and their criminal origin through complex transaction chains. Blockchain forensics is most effective at tracing and documenting layering activity, as every hop is permanently on-chain.
Lazarus Group
North Korea's elite state-sponsored hacking unit, controlled by the Reconnaissance General Bureau (RGB). Stole $2B+ in crypto in 2025 including the $1.5B Bybit hack. Uses crypto theft to fund DPRK's nuclear weapons program. Sub-groups: TraderTraitor, Contagious Interview, APT38.
MaaS (Malware-as-a-Service)
The sale of malware (especially infostealers and ransomware) on subscription or affiliate models via darknet marketplaces. Enables non-technical criminals to deploy sophisticated malware. RaaS (Ransomware-as-a-Service) is the ransomware-specific variant.
Mempool
The pool of unconfirmed blockchain transactions awaiting inclusion in the next block. Investigators can monitor the mempool for real-time transaction alerts — critical for rapid response to exchange hacks when funds are still moving.
Mixer / Tumbler
Services pooling funds from multiple users and redistributing them to break transaction chains. Examples: Tornado Cash (ETH, OFAC-sanctioned 2022), Wasabi Wallet (BTC CoinJoin). Using a mixer does not make funds untraceable — it creates a fork requiring parallel output analysis.
MLAT
Mutual Legal Assistance Treaty. International agreement enabling countries to share legal process and evidence. Used to obtain records from VASPs in foreign jurisdictions. Speed varies significantly by country — some jurisdictions have streamlined crypto-specific cooperation channels.
Money Mule
An individual who receives and transfers illicit proceeds — knowingly or unwittingly — on behalf of a criminal network. In crypto: account mules (open exchange accounts), transfer mules (forward crypto), and cash-out mules (convert to fiat). The KYC holder is often the mule, not the criminal.
Monero (XMR)
A privacy-focused cryptocurrency using ring signatures and stealth addresses to obscure sender, receiver, and amount. The hardest blockchain to trace. Preferred by ransomware groups for payments. Exchange on/off-ramps remain investigable chokepoints.
NFT Wash Trading
Buying NFTs between self-controlled wallets at artificially inflated prices to establish "legitimate" art market transaction records for funds. A placement and layering tactic creating apparent legitimate provenance.
OFAC
U.S. Office of Foreign Assets Control. Sanctions crypto wallet addresses and services linked to sanctioned entities — Lazarus Group wallets, Tornado Cash, Sinbad.io, ChipMixer. Transactions with OFAC-listed addresses create legal liability for VASPs in any jurisdiction.
OSINT
Open Source Intelligence. Information gathered from publicly available sources — social media, WHOIS records, court filings, breach databases, and forum posts. Combined with on-chain blockchain data, OSINT bridges the gap from pseudonymous wallet to real-world identity.
OTC Broker
Over-the-counter broker facilitating large crypto-fiat conversions outside formal exchange infrastructure. The most common final cashout mechanism for major criminal networks. Some are licensed MSBs; others are entirely unlicensed. Documented in the Khinkis ransomware case (March 2026).
Pass-Through Address
An address that receives funds and immediately forwards the entire amount, leaving a $0 balance. A significant red flag for layering — the controller paid transaction fees for an economically unnecessary step. Documented in TRM Labs' Flip Book as a primary illicit activity indicator.
PCMLTFA
Proceeds of Crime (Money Laundering) and Terrorist Financing Act — Canada's primary AML/CFT legislation. Establishes reporting requirements for VASPs (including crypto exchanges) operating in Canada. FINTRAC is the competent authority.
Peeling Chain
A layering technique sending funds through many sequential addresses, each forwarding slightly less than the prior — creating a long trail to exhaust investigative resources. Commercial tools like TRM Labs trace these automatically.
Phishing
Deceptive communications impersonating trusted entities to steal credentials, seed phrases, or private keys. Variants: spear-phishing (targeted), whale phishing (high-value targets), and vishing (voice phishing). The initial access vector for the $1.5B Bybit hack.
Pig Butchering (Sha Zhu Pan)
A long-con romance/investment fraud originating from Southeast Asian cyber-scam compounds. Criminals cultivate trust over months before directing victims to fraudulent crypto investment platforms. The most financially devastating fraud category by total losses. Primarily uses TRON USDT for cashout.
Placement
The first stage of money laundering — introducing illicit funds into the financial system. The highest-risk stage for detection. Crypto placement methods: Bitcoin ATMs, P2P exchanges, structuring, and casino chips. Also the least blockchain-visible stage as it often starts with cash.
Private Key
The cryptographic secret that controls a cryptocurrency wallet. Whoever holds the private key controls the funds. Never share it. Seed phrase compromise (12/24 words) is equivalent to private key compromise for HD wallets.
Pseudonymous
Not fully anonymous. Blockchain addresses contain no names but are traceable through behavioral patterns, VASP chokepoints, and OSINT correlation. "Pseudonymous ≠ anonymous" is the fundamental principle of blockchain investigation.
RaaS (Ransomware-as-a-Service)
The business model for major ransomware groups: developers provide the malware and infrastructure; affiliates deploy attacks; proceeds split (typically 20-30% developer / 70-80% affiliate). Examples: LockBit, ALPHV/BlackCat, Cl0p.
Ransomware
Malware encrypting victim files/systems and demanding cryptocurrency for the decryption key. Modern variants use double extortion (encrypt + threaten to publish data). Payments in BTC or XMR create on-chain evidence trails for investigators.
Rug Pull
A DeFi fraud where developers drain a token's liquidity pool and abandon the project after attracting investor funds. Hard rug = immediate drain; soft rug = gradual abandonment. Common on low-barrier DEX platforms (Uniswap, PancakeSwap, Pump.fun).
SAR (Suspicious Activity Report)
Report filed by financial institutions and VASPs when suspicious transactions are detected. In Canada: Suspicious Transaction Report (STR) filed with FINTRAC. In the US: SAR filed with FinCEN. A common source of investigative leads for crypto cases.
Seed Phrase
A 12 or 24-word mnemonic that provides complete access to all wallets derived from an HD wallet. Equivalent to the private key — compromise of the seed phrase means total, irreversible loss of all associated wallet funds.
Sextortion
Extortion using threats to distribute intimate images unless cryptocurrency payment is made. Variants: mass email campaigns using breach passwords as "proof" of webcam access; actual intimate image theft; AI-generated deepfake imagery as leverage.
SIM Swap
Fraudulent transfer of a victim's mobile phone number to an attacker-controlled SIM card — enabling interception of SMS 2FA codes. Used to bypass two-factor authentication on cryptocurrency exchange accounts to facilitate account takeover.
Smart Contract
Self-executing code on a blockchain automatically enforcing agreement terms. DeFi protocols run on smart contracts. Exploits target smart contract logic errors. All contract interactions are on-chain and traceable; the contract deployer wallet is always investigable.
Spear-Phishing
Targeted phishing attack against a specific individual, researched in advance for maximum credibility. Used by Lazarus Group (Contagious Interview) against crypto exchange developers. The initial access vector for multiple multi-hundred-million-dollar hacks.
Straw Identity
A person used to open an account or provide KYC credentials on behalf of the real criminal — inserting a false identity layer. Common in mule networks and exchange account fraud. The KYC holder at a VASP is often a straw identity; always investigate beyond the named account holder.
Structuring
Breaking large amounts into multiple sub-reporting-threshold transactions to avoid triggering mandatory AML reporting. A criminal offense in itself in most jurisdictions, independent of the underlying predicate crime. Detectable through transaction pattern analysis.
Supply Chain Attack
Compromising a target indirectly by attacking a trusted third-party supplier or tool. The Bybit $1.5B hack began with a supply chain compromise of Safe{Wallet} — a developer machine infection that allowed malicious JavaScript injection into the signing interface.
Timing Analysis
Cross-referencing exact blockchain transaction timestamps against external events — social media posts, exchange login logs, real-world activity — to correlate identities or confirm behavioral patterns. Used by ZachXBT in multiple landmark investigations.
Tornado Cash
An Ethereum-based cryptocurrency mixing protocol sanctioned by OFAC in August 2022. Lazarus Group's preferred ETH mixing tool post-hack. Developer Roman Storm convicted in 2024. The sanctions established that smart contract addresses themselves can be designated as sanctioned entities.
Transaction Hash (TXID)
A unique alphanumeric identifier for a specific blockchain transaction — the digital receipt number. Used to look up any transaction on a block explorer. The anchor for any on-chain investigation. Best practice: copy electronically to avoid transcription errors.
VASP (Virtual Asset Service Provider)
Any platform used to buy, sell, trade, or exchange virtual currency. The FATF term covering all crypto exchanges and custodians. Most VASPs collect KYC data that investigators can obtain via legal process. The primary target for legal process in cryptocurrency investigations.
Wallet
Software or hardware tool for storing cryptocurrency private keys. Technically, the funds are on-chain — the wallet stores the key that controls them. Non-custodial wallets (MetaMask, Ledger) have no central operator; custodial wallets (exchange accounts) have VASPs who hold KYC data.
Whale Phishing
Highly targeted spear-phishing against extremely high-value individuals — exchange executives, DeFi protocol administrators, or crypto-wealthy individuals. A single successful whale phishing attack can yield tens or hundreds of millions of dollars.

DFIR & Enterprise Threat Hunting

Digital Forensics and Incident Response meets E-Crime investigation. This module covers the critical pre-on-chain phase — endpoint forensics, memory analysis, mobile forensics, and enterprise threat hunting — that determines how a crypto-stealer, infostealer, or ransomware payload was deployed before any funds moved on-chain.

🔗 Why DFIR Matters in E-Crime
On-chain tracing shows where stolen funds went. DFIR answers how the attacker got in — the initial access, persistence, execution, and credential harvesting phases that preceded the financial crime. Without DFIR, investigations may trace funds to a dead end (a mule account) and never attribute the root cause actor. The complete picture requires both.

Endpoint forensics reconstructs attacker activity on a compromised machine by examining the digital artifacts left behind — file system metadata, registry hives, event logs, prefetch data, and browser artifacts. In e-crime investigations, this establishes the complete attack timeline: from initial phishing delivery through credential harvest, lateral movement, and final payload execution.

The critical investigation question: was the victim machine's crypto wallet directly accessed? Were exchange session cookies stolen? Was a clipboard hijacker installed? These questions are answered through endpoint forensics — not blockchain tracing.

PHASE 1
Evidence Preservation
Before analysis, preserve evidence integrity: create verified forensic images of all drives (FTK Imager, dd), capture volatile memory (RAM) immediately — before shutdown — using Magnet RAM Capture or WinPmem. Document chain of custody. Never analyze live evidence directly.
PHASE 2
Triage & Scoping
Rapid triage to scope the incident: run Kroll Artifact Parser and Extractor (KAPE) or Velociraptor for targeted artifact collection. Identify compromised accounts, affected hosts, and approximate compromise timeline before committing to full forensic acquisition on every endpoint.
PHASE 3
Timeline Construction
Aggregate file system MAC timestamps, event log entries, prefetch execution records, browser history, and registry modification times into a unified timeline using tools like Plaso/log2timeline or Timesketch. The timeline maps attacker dwell time and activity sequence.
PHASE 4
Artifact Deep-Dive
Investigate key artifacts per platform (see Artifacts tab). Focus on: what was executed, what was accessed, what was exfiltrated, and what persistence mechanisms were installed.
Windows — Execution Artifacts
  • Prefetch (.pf): Records of executed programs with timestamps and run count — confirms malware execution even if binary is deleted
  • Shimcache / AppCompatCache: Registry artifact recording executable metadata; tracks binaries that have been present on the system
  • Amcache.hve: SHA-1 hashes of executed programs — enables malware identification even post-deletion
  • UserAssist: Registry keys recording GUI program execution with timestamps and run counts
  • BAM/DAM: Background Activity Monitor — records background process execution with timestamps
Windows — Persistence Locations
  • Registry Run keys: HKCU\Software\Microsoft\Windows\CurrentVersion\Run
  • Scheduled Tasks: C:\Windows\System32\Tasks\ — XML task definitions
  • Startup folders: %APPDATA%\Microsoft\Windows\Start Menu\Programs\Startup
  • Services: HKLM\SYSTEM\CurrentControlSet\Services — malware as service
  • WMI Subscriptions: Fileless persistence mechanism — check with Get-WMIObject
Windows — Event Logs
  • 4624/4625: Successful/failed logon — reveals unauthorized access attempts
  • 4688: Process creation with command line (if auditing enabled) — reveals malware spawning
  • 4698/4702: Scheduled task created/modified
  • 7045: New service installed — ransomware/RAT persistence
  • 4663: Object access — file reads on wallet.dat or seed phrase files
  • Sysmon (1,3,7,8,10): Process create, network connect, image load, CreateRemoteThread, ProcessAccess — if Sysmon deployed
Browser & Credential Artifacts
  • Chrome/Edge Login Data: Saved password database (SQLite) at %LOCALAPPDATA%\Google\Chrome\User Data\Default\Login Data — infostealers target this directly
  • Cookie databases: Session cookies for exchange logins — stolen cookies enable session hijacking without password
  • MetaMask vault: %APPDATA%\Local\Google\Chrome\User Data\Default\Local Extension Settings\nkbihfbeogaeaoehlefnkodbefgpgknn — encrypted vault file
  • Clipboard history: Windows clipboard history (Win+V) may contain copied wallet addresses or seed phrases
Linux & macOS Artifacts
  • bash_history / zsh_history: Command execution history — reveals attacker commands and tool usage
  • Cron jobs: /etc/cron*, /var/spool/cron — persistence mechanism for cryptojacking and backdoors
  • Systemd services: /etc/systemd/system/ — malware installed as systemd unit
  • macOS LaunchAgents/Daemons: ~/Library/LaunchAgents — common macOS malware persistence
  • macOS Keychain: /Library/Keychains/ — contains stored passwords including exchange credentials
  • auditd / syslog: Authentication events and process execution on Linux
Network Artifacts
  • PCAP / NDR logs: C2 communication patterns, data exfiltration volumes, DNS queries to malicious domains
  • DNS cache: ipconfig /displaydns — recently resolved domains may reveal C2 infrastructure
  • Firewall / proxy logs: Outbound connections to mining pools (stratum protocol), paste sites, or unusual TLDs
  • NetFlow data: Volume-based anomaly detection for large exfiltration events
Crypto Wallet Files to Recover
  • wallet.dat — Bitcoin Core wallet file; contains encrypted private keys
  • MetaMask vault — Chrome extension encrypted vault at Local Extension Settings path
  • Exodus/Electrum config directories — %APPDATA%\Exodus, %APPDATA%\Electrum\wallets
  • Ledger Live app data — account metadata (not private keys, which stay on hardware device)
  • Clipboard history artifacts — may contain seed phrases or private keys if victim copied them
Infostealer Indicators on Disk
  • Suspicious DLL sideloading from user-writable paths (%TEMP%, %APPDATA%)
  • AutoIt or NSIS-compiled binaries in Downloads or Temp
  • Unusual PowerShell execution (base64-encoded commands, AMSI bypass)
  • Evidence of SQLite database access on Chrome profile paths
  • Outbound ZIP/archive creation in %TEMP% followed by network upload
Ransomware Forensic Indicators
  • VSS deletion commands: vssadmin delete shadows /all /quiet
  • bcdedit /set {default} recoveryenabled No — disables recovery
  • wbadmin delete catalog — removes backup catalog
  • Encrypted file extensions and ransom note drops in enumerated directories
  • Network share enumeration (net view, SMB scanner) before encryption
Connecting Endpoint to On-Chain
The DFIR pivot to blockchain investigation: establish the timestamp when credentials/keys were stolen from the endpoint → compare to the timestamp of the unauthorized on-chain transaction. Correlation establishes that the endpoint compromise caused the on-chain theft — critical for legal proceedings and insurance claims.

CriticalVolatile Evidence
›
Why Memory
Analysis Workflow
Crypto Findings
Tools

Memory forensics examines the contents of a system's RAM — the most forensically rich and most ephemeral evidence source. RAM contains artifacts that never touch disk: decrypted malware payloads, injected shellcode, C2 communication keys, plaintext credentials, browser session tokens, and — critically for crypto investigations — decrypted wallet private keys and seed phrases that an infostealer was in the process of exfiltrating.

⚡ Time-Critical
RAM evidence is destroyed when power is cut. Memory acquisition must happen before system shutdown. In ransomware incidents: capture memory before removing from network. The encryption key used by the ransomware may be recoverable from RAM — enabling decryption without paying ransom.

Fileless malware — increasingly common among APTs including Lazarus Group — exists only in memory. There is no malicious file on disk to recover. Memory forensics is the only way to capture these artifacts.

1
Acquisition
Capture RAM using WinPmem, Magnet RAM Capture, or LiME (Linux kernel module). Output: raw memory dump file. Verify integrity with hash. On cloud instances: use snapshot functionality (AWS EC2 hibernation, Azure memory dump) for IaaS forensics.
2
Profile Identification
Determine the OS version and kernel structures for the memory image — required for Volatility to correctly parse process structures. Use vol.py -f memory.dmp imageinfo or Volatility3's automatic symbol resolution.
3
Process Analysis
Run pslist, pstree, and psscan to enumerate processes — including hidden/unlinked ones that rootkits hide from pslist. Cross-reference parent-child relationships: cmd.exe spawned by Word.exe is suspicious. Identify processes with no disk-resident binary (dlllist on orphaned PIDs).
4
Injection Detection
Use malfind to identify memory regions with MZ/PE headers (injected PE files) and executable permissions in unexpected locations. hollowfind detects process hollowing. cmdline reveals command line arguments of all processes including obfuscated PowerShell.
5
Network Artifacts
netscan / connections — reveals all open and recently closed network connections, including those to C2 infrastructure. Active HTTPS connections may reveal C2 domain even if DNS was DoH-based. Correlate PIDs to processes.
6
Credential & Key Extraction
hashdump — extracts NTLM password hashes from LSASS memory. mimikatz-equivalent modules recover cleartext credentials cached in LSASS (WDigest). String scanning for wallet addresses, seed phrase word patterns, and private key formats (0x + 64 hex chars) within browser and wallet process memory.
Private Key Recovery
Scan process memory of wallet applications (Bitcoin Core, Exodus, MetaMask browser extension) for private key patterns: 64-character hex strings, WIF format (starts with 5, K, or L), or 256-bit random number patterns. Keys may be briefly in memory during signing operations.
Seed Phrase Detection
Scan memory for BIP-39 word list sequences (2048 words) in runs of 12 or 24 consecutive words. Victims who type seed phrases into a compromised machine leave them in keyboard buffer memory, browser process memory, and input handler memory briefly.
Ransomware Key Recovery
Ransomware encryption keys are generated in memory before files are encrypted. If memory is captured early enough in the encryption process, symmetric keys (AES-256) may be recoverable using pattern matching on known key schedules. This approach recovered keys in early WannaCry variants.
Browser Session Token Theft
Session cookies for exchange accounts (Coinbase, Binance) are stored in browser process memory. Infostealers target Chrome's memory space specifically to extract session tokens without needing the account password. Dump Chrome.exe memory and search for "remember_token" and "__Secure-" cookie patterns.
Infostealer C2 Config
Infostealers decrypt their C2 configuration from memory at runtime. The decrypted config — containing C2 domains, exfiltration endpoints, and targeting rules — may be recoverable via memory scanning of the malicious process even if on-disk it is encrypted.
Clipboard Contents
Clipboard hijacker malware monitors the Windows clipboard API. The clipboard buffer — containing whatever the victim last copied — is readable from the win32k.sys memory space. May contain recently copied wallet addresses, private keys, or passwords.

Mobile devices are the primary platform for cryptocurrency management — most users manage wallets, access exchanges, and authenticate via SMS 2FA exclusively on mobile. Mobile forensics is therefore essential in: SIM swap investigations (carrier and handset evidence), exchange ATO cases (session tokens and app data), pig butchering (victim communication records), and mobile banking malware investigations.

Mobile forensics presents unique challenges: hardware encryption (iOS Secure Enclave, Android StrongBox), device locking, cloud backups that may preserve deleted data, and a rapidly evolving acquisition landscape as vendors patch forensic extraction methods.

Acquisition Levels
  • Logical: Extracts accessible files and databases via OS APIs — fastest, least invasive, but limited by device lock state
  • File System: Full file system extraction — requires jailbreak (iOS) or rooting (Android) or exploit-based acquisition
  • Physical: Bit-for-bit device image — most complete; requires chip-off or advanced exploit (UFED, GrayKey)
  • Cloud: iCloud/Google Account backup extraction — often the most practical when device is inaccessible
Critical Evidence Types
  • SMS messages (SIM swap evidence, 2FA codes)
  • Communication apps (Telegram, WhatsApp, Signal)
  • Exchange app data (Coinbase, Binance, Kraken)
  • Crypto wallet apps (Trust Wallet, MetaMask Mobile, Exodus Mobile)
  • Browser history and cookies
  • Photos with EXIF geolocation data
  • Call records and contacts
  • Installed application list
iOS Acquisition Methods
  • iTunes/Finder Backup: Encrypted backup contains app data, messages, keychain items. Set backup password to unlock keychain. Unencrypted backup excludes keychain.
  • iCloud Backup: Requires Apple ID credentials or court-ordered Apple data preservation request. Contains most backup data excluding end-to-end encrypted items.
  • Advanced Logical (Checkm8): Exploit-based acquisition for A5–A11 chip devices. Provides file system access without jailbreak. Tools: Cellebrite UFED, Elcomsoft iOS Forensic Toolkit.
  • GrayKey: Law enforcement device enabling passcode brute-force/bypass on supported iOS versions. Access varies significantly by iOS version — recent versions limit effectiveness.
Key iOS Forensic Locations
  • Keychain: Stores exchange credentials, app passwords, and crypto wallet seeds (for apps using Keychain for storage)
  • Messages.db: iMessage and SMS history — pig butchering communications, SIM swap confirmation codes
  • KnowledgeC.db: Screen time and app usage records with timestamps — establishes when exchange apps were accessed
  • BiomeDB: User activity analytics database containing app interactions and durations
  • Installed apps: /var/containers/Bundle/Application/ — app bundles and container data
iOS Crypto App Data
  • Trust Wallet: Wallet data in app sandbox — keystore files if device compromised
  • Coinbase app: Authentication tokens, account data, transaction history in app container
  • MetaMask Mobile: Encrypted vault in app sandbox — passphrase required for decryption; may be recoverable if device has PIN access
  • Safari/Chrome: Browser history shows exchange logins; autofill may contain exchange credentials
iOS Legal Process
  • Apple Law Enforcement Guidelines: apple.com/legal/privacy/law-enforcement-guidelines-us.pdf
  • iCloud data preservation requests via Apple's law enforcement portal
  • Carrier records via separate subpoena for SIM swap evidence
  • Apple provides: subscriber info, iCloud content (with order), device backups, financial transactions, app data for specified apps
Android Acquisition Methods
  • ADB (Android Debug Bridge): Logical extraction via USB — requires USB debugging enabled. Collects installed apps and accessible data but limited by encryption.
  • Android Backup Protocol: Full backup via adb backup — app-dependent; many apps disable backup via android:allowBackup=false
  • EDL Mode (Qualcomm): Emergency Download Mode — low-level extraction for Qualcomm devices. Most powerful Android acquisition method for supported chipsets.
  • JTAG / Chip-off: Physical acquisition methods — destructive; used when device is damaged or PIN-locked with no other option
  • Google Account Cloud: Google Takeout / law enforcement request — contacts, location history, Gmail, Drive backups
Key Android Forensic Locations
  • /data/data/[package]/: Application private data directory — contains databases, shared preferences, and cached files for each app
  • SMS Provider: /data/data/com.android.providers.telephony/databases/mmssms.db — all SMS including 2FA codes
  • Google Chrome History: /data/data/com.android.chrome/app_chrome/Default/History — SQLite database
  • Call logs: /data/data/com.android.providers.contacts/databases/calllog.db
  • Installed packages: /data/app/ and PackageManager.db
Android Crypto App Artifacts
  • Trust Wallet: /data/data/com.wallet.crypto.trustapp/files/ — keystore and wallet files
  • MetaMask Android: /data/data/io.metamask/ — encrypted vault data
  • Exchange Apps: Authentication tokens in SharedPreferences XML files — may persist session tokens used in ATO
  • Accessibility Service Abuse: Banking malware on Android abuses Accessibility Services — check /data/data/[malware package]/shared_prefs/ for harvested data
Android Malware Persistence
  • Device Admin privileges — makes uninstallation difficult without factory reset
  • Accessibility Service registration — persists through most user actions
  • Boot receivers (BOOT_COMPLETED) — restarts on every device reboot
  • APK installed from unknown sources with excessive permissions
Mobile Crypto Wallet Forensics
  • Recover encrypted wallet files from app sandboxes — device passcode or app password required for decryption
  • Search for seed phrase fragments in autofill databases, Notes app, and keyboard cache (SwiftKey, Gboard learn-by-typing caches)
  • Photos app may contain photographed seed phrase backup sheets with EXIF timestamp
  • Clipboard history on Android 10+ stores recent clipboard contents accessible forensically
SIM Swap Evidence
  • SMS database: 2FA codes received immediately before ATO event establish timeline
  • Carrier portal logs: who called to request SIM transfer, what account credentials provided
  • Device IMEI and ICCID mismatch with expected values after swap
  • Baseband logs: network registration events showing new SIM activation timestamp
Mobile Banking Malware Artifacts
  • Accessibility Service logs showing apps targeted by overlay attacks
  • Screen recording malware artifacts: video files in hidden directories
  • Keylogger data stored in app private directory before exfiltration
  • Fake exchange overlay APKs installed alongside legitimate apps
Pig Butchering Communication Evidence
  • Telegram, WhatsApp, and WeChat message databases — document the full social engineering timeline
  • Dating app messages (Tinder, Hinge, Bumble) — initial contact by scammer
  • Photos shared by scammer (reverse image search for fake identity)
  • Transaction records showing funds sent to scammer-directed wallets

Threat hunting is proactive adversary discovery — searching enterprise telemetry for evidence of compromise that has evaded automated detection. Unlike DFIR (reactive investigation of a known incident), threat hunting assumes the environment is already compromised and systematically seeks evidence.

For e-crime hunting, the investigative focus differs from generic threat hunting: the goal is specifically to identify the tradecraft preceding financial crime — credential theft infrastructure, wallet-targeting malware, exchange session hijacking, and insider threat behavior involving crypto asset systems.

Hunt Loop Methodology
  • Hypothesis: Form a threat-driven hypothesis (e.g., "Lazarus Group is targeting our exchange developer machines via LinkedIn spear-phishing")
  • Data Collection: Identify telemetry sources that would contain evidence if the hypothesis is true (EDR process telemetry, email gateway logs, browser proxy logs)
  • Investigation: Query collected data for indicators matching the hypothesis
  • Response: Escalate confirmed findings; update detection rules; document IOCs
MITRE ATT&CK Mapping
Map all hunt hypotheses to MITRE ATT&CK tactics: Initial Access (T1566 Phishing) → Execution (T1059 Command Line) → Persistence (T1547 Registry Run Keys) → Credential Access (T1555 Credentials from Password Stores) → Collection (T1005 Data from Local System) → Exfiltration (T1041 Exfiltration Over C2). Use ATT&CK Navigator to track coverage gaps.
Hunting Infostealer Deployment
  • Hunt for SQLite database access on Chrome profile paths by non-browser processes
  • PowerShell executing from %TEMP% or %APPDATA% with encoded commands
  • Processes reading DPAPI-protected browser credential blobs
  • Outbound ZIP archive uploads to uncommon cloud storage domains
  • AutoIt (AutoIt3.exe) or NSIS stub execution from user-writable paths
Hunting Ransomware Pre-Encryption
  • vssadmin.exe, wmic.exe, or bcdedit.exe execution by non-admin processes
  • Cobalt Strike default beacon indicators: rundll32 loading from %TEMP%, unusual parent-child process chains
  • PsExec or WMIC lateral movement — unusual SMB authentication patterns
  • Large-scale file rename operations (file extension changes) in monitored directories
  • Kerberoasting: unusual TGS requests for service accounts from workstations
Hunting Cryptojacking
  • Outbound connections to mining pool domains or IPs (port 3333, 4444, 14444 — Stratum protocol)
  • Sustained CPU >80% without matching user workload — alert on EDR CPU telemetry
  • Cron jobs or scheduled tasks executing from /tmp or %TEMP% paths
  • Docker containers with excessive CPU allocation or mining pool network connections
  • Cloud billing alerts: unexpected compute spend in AWS/GCP/Azure
Hunting BEC Infrastructure
  • Email gateway: typosquatting domain registrations similar to corporate domain (levenshtein distance 1-2)
  • OAuth application consent grants to unfamiliar third-party apps (MFA bypass method)
  • Inbox rules created via EWS/Graph API forwarding emails to external addresses
  • Unusual email send volume from single accounts — compromised mailbox used as spam relay
  • Login from impossible travel (same account, two geographies within minutes)
Hunting Insider Crypto Theft
  • Privileged user accessing crypto custody wallet systems outside business hours
  • Database queries against KYC or wallet management systems from unusual endpoints
  • Large file downloads from crypto-related internal systems (cold wallet management tools)
  • Screen recording or clipboard monitoring utilities installed on privileged workstations
  • Anomalous VPN usage: admin connecting from residential IP during off-hours (USMS Daghita case pattern)
Hunting Supply Chain Compromise
  • Third-party software update processes making unexpected network connections
  • JavaScript injection in web-served financial application files (Bybit attack pattern)
  • Hash mismatches on deployed application files vs. known-good baseline
  • Unexpected outbound connections from developer tooling (npm, pip, cargo) to unknown domains
  • CI/CD pipeline logs showing unexpected environment variable access or secret exfiltration

1
Hash & Reputation Check
Hash the sample (MD5/SHA-256). Search VirusTotal, MalwareBazaar, and ANY.RUN. Many infostealers are well-documented — existing community analysis may immediately provide C2 domains and behavioral IOCs, saving hours of manual work.
2
File Identification
Determine true file type with file + Detect-It-Easy (DIE) — ignore the extension. Identify: PE format (32/64-bit), packer or protector (UPX, custom), section entropy (high = packed/encrypted content), compiler artifact (Golang, .NET, C++, Rust). Compiler choice determines your RE approach.
3
String Extraction
Run strings -n 8 (ASCII) and strings -e l (Unicode). Search for: hardcoded URLs/IPs, registry keys, file paths, error messages, wallet address regex patterns, Telegram bot tokens, and Discord webhook URLs. Many infostealers have partially-visible config strings before unpacking.
4
Automated Sandbox
Submit to Hybrid Analysis or ANY.RUN (interactive). Review: spawned processes, network connections (C2 IP/domain), files dropped, registry changes, and API call sequence. Sandbox output often provides the C2 domain and dropper chain within minutes — always do this before manual RE.
5
Unpack if Needed
Identify packer with DIE. UPX: upx -d sample.exe. Custom packers: set OEP (Original Entry Point) breakpoint in x64dbg — when execution transfers to OEP, dump unpacked PE from memory with Scylla. Rebuild imports from the dumped image before analysis.
6
C2 Config Extraction
Locate and decrypt the C2 configuration — the key investigative output. Methods: static decryption (find XOR/AES key, write Python script), dynamic extraction (run to config decryption in debugger, dump plaintext from output buffer), or use a published config extractor if the family is identified.
Import Table Analysis

  • RegSetValue → registry persistence

  • Control flow flattening: Obfuscator-LLVM pattern — all paths through dispatcher switch
  • Junk code: Dead code inserted between real instructions to slow analysis

Mobile Infostealers

Infostealers follow a consistent architectural pattern regardless of family: Deploy → Enumerate Targets → Harvest Credentials → Package → Exfiltrate → Self-Delete. Recognizing each phase in disassembly accelerates analysis of any new family.

Deployment Chains
  • Phishing email → malicious Office macro → PowerShell downloader → stealer PE
  • Pirated software installer (NSIS/InnoSetup) bundling stealer payload
  • YouTube crack/keygen video → ZIP → AutoIt loader → stealer
  • Malicious npm/pip package executing stealer during install
  • Malvertising → drive-by download → JS dropper → stealer
Enumeration Targets
  • Browser profile directories (Chrome, Firefox, Brave, Edge — all profile paths)
  • Crypto wallet app directories: Exodus (%APPDATA%\Exodus), Electrum (%APPDATA%\Electrum\wallets), Atomic Wallet
  • Browser extension storage: MetaMask, Phantom (Local Extension Settings paths by extension ID)
  • Desktop/Documents files matching *.txt, *.kdbx, *.wallet, *seed*, *private*
  • System fingerprint: OS, username, timezone, installed AV, screenshots
DPAPI Chrome Credential Theft
Chrome encrypts passwords with Windows DPAPI tied to the current user session. Infostealers call CryptUnprotectData() as the victim user — decrypts Chrome's Login Data SQLite database without any master password. In disassembly: identify SQLite3 library calls on the Login Data path, followed by CryptUnprotectData on the "password_value" column bytes.
Exfiltration Methods
  • HTTP POST to C2 panel (most common — look for multipart/form-data with ZIP archive)
  • Telegram Bot API: hardcoded bot token + chat ID — extract both as IOCs
  • SMTP to attacker email (older families)
  • Discord webhook URL (common in low-sophistication stealers — easy to burn)
  • FTP upload to attacker server
RedLine Stealer (.NET)
  • Decompile with dnSpy — near-source C# output
  • Config encrypted with AES-128; key hardcoded or derived from mutex name
  • C2 endpoint stored as IP:port string in config class
  • Targets: browser creds, crypto wallets, FileZilla, Discord tokens, Steam sessions, VPN configs
  • Community config extractors available on GitHub — run before manual analysis
Vidar / Lumma Stealer (C++)
  • Use IDA Pro/Ghidra + x64dbg for combined static/dynamic analysis
  • C2 domain retrieved from attacker-controlled public profiles (Steam bio, Telegram channel) at runtime — evades static domain blocklists
  • Hardcoded "gate" URL in binary; gate returns active C2 list on each run
  • Strong anti-VM: CPUID checks, RDTSC timing, mouse cursor movement verification — use Pafish to identify all checks, then patch each
Racoon Stealer (C/C++)
  • Telegram channel used as dead-drop for C2 IP address at runtime
  • Config contains Telegram channel ID — query channel to observe current C2 IP
  • SQLite and NSS libraries embedded statically for browser credential parsing
  • Generates unique machine ID from hardware fingerprint for victim tracking
Golang Infostealers
  • Golang binaries are statically linked and large (~10MB+) — normal for Go
  • Function names often stripped in release builds; use go-re Ghidra plugin to recover Go runtime function names
  • String table is separate from code — extract with gostrings for Go-specific string extraction
  • HTTP client implementation is often standard library — easier to identify C2 communication functions
1
Locate Decryption Routine
Search Ghidra/IDA for AES S-box constants (0x63637c7c77...) or XOR loops operating on byte arrays. Set breakpoint at the routine's return instruction in x64dbg. When hit, the output register/buffer contains the decrypted configuration in plaintext.
2
Extract All C2 Indicators
From decrypted config extract: C2 domain(s)/IP(s) and port, panel URL paths (/gate.php, /loader/tasks), bot identifier format, campaign tag, and target application list. Document all as network-level IOCs for SIEM detection rules and hunting queries.
3
Infrastructure Pivot
For each C2 domain: run WHOIS, passive DNS (VirusTotal, SecurityTrails), SSL certificate history (crt.sh), and Shodan banner scan. Identify all other domains and IPs on the same infrastructure — often reveals other malware campaigns operated by same threat actor.
4
Telegram Bot Pivot
Extract bot token (format: [ID]:[HASH]) and chat ID. Query api.telegram.org/bot[TOKEN]/getChat?chat_id=[ID] — may reveal group name, member list, or linked channels. Submit token to Telegram abuse portal for takedown. The chat ID may resolve to a public channel enabling victim count estimation.
5
Write Config Extractor
Automate decryption for future samples of the same family with a Python extractor script. Publish to MalwareBazaar or community GitHub. A 50-line extractor saves hours per future sample and enables large-scale C2 hunting across malware corpora.
Keylogger Hook Analysis
  • Identify SetWindowsHookEx(WH_KEYBOARD_LL, ...) call — the low-level keyboard hook API
  • Locate the hook callback function (LowLevelKeyboardProc) — this is where keystrokes are processed
  • Find the log buffer: where are keystrokes accumulated? In-memory string, registry value, or %TEMP% file?
  • Identify exfiltration trigger: time-based (timer callback), size-based (buffer threshold), or event-based (window focus change)
  • Target-aware variant: GetForegroundWindow() + string comparison to exchange domain name — only logs when exchange tab is focused
Clipboard Hijacker Analysis
  • Identify AddClipboardFormatListener() or SetClipboardViewer() — clipboard monitoring APIs
  • Locate the WM_CLIPBOARDUPDATE message handler — this runs on every clipboard change
  • Find pattern matching logic: regex or character-length checks for crypto address formats (Bitcoin: 26-35 base58 chars; Ethereum: 0x + 40 hex; Monero: 95 chars starting with 4)
  • Extract hardcoded replacement addresses — these are the on-chain investigation anchor: trace them directly on the blockchain

  • Permissions audit: legitimate wallet apps need minimal permissions; fakes request SMS, Contacts, Camera, Accessibility
SMS Interception Malware

  • Trigger conditions: some only forward SMS matching regex patterns (OTP, verification code keywords)
  • iOS: legitimate apps cannot read SMS — iOS SMS interception requires MDM profile or carrier-level interception; investigate at infrastructure level

Standard Hybrid Encryption
Most modern ransomware: AES-256 encrypts file contents (symmetric, fast); the AES key itself is encrypted by an RSA-2048/4096 public key embedded in the malware binary. The attacker holds the RSA private key server-side — mathematical decryption without it is infeasible. Understanding this scheme tells you whether key recovery is theoretically possible.
Modern Variants (Curve25519/ChaCha20)
LockBit 3.0, ALPHV/BlackCat (Rust), and newer families use Curve25519 ECDH for key encapsulation with ChaCha20 stream cipher for file encryption. In Ghidra: search for the "expand 32-byte k" ChaCha20 constant. Rust-based samples use the ring crypto crate — identify ring:: function call patterns.
Implementation Weaknesses to Hunt
  • Weak PRNG (time-seeded srand()) → key space brute-forceable from infection timestamp
  • Same IV reused across files → ciphertext comparison attacks possible
  • Partial file encryption (first/last N bytes only) → partial file recovery possible with header reconstruction
  • Key temporarily written to disk → forensic recovery from pagefile or disk slack
Ransom Note as Evidence
  • Hash and submit to VirusTotal for instant family identification
  • Victim ID format encodes campaign/affiliate data — document for group attribution
  • TOR payment portal URL: unique per operation; submit to threat intel feeds
  • Language artifacts (grammar, idioms) may indicate operator nationality
1
Locate Crypto Constants
In Ghidra: Search → Scalar search for AES S-box (0x63), ChaCha20 constant bytes ("expand 32-byte k" = 65 78 70 61...), or SHA-256 initial hash values. CAPA auto-flags cryptographic capability. These anchor points lead to the key derivation and encryption logic.
2
Trace Key Generation Flow
From program entry, follow execution to the first encryption call. Map the key generation path: Is the symmetric key from CryptGenRandom / BCryptGenRandom (strong) or srand(time()) (weak)? How is it encapsulated — RSA public key encrypt, ECDH key agreement? Where is the encrypted key written — ransom note footer, .key file, C2 upload?
3
Analyze File Targeting & Exclusion Logic
Identify: target extension list (whitelist or blacklist approach), excluded directories (Windows, ProgramFiles — prevents system destruction), encryption mode (full / header-only / intermittent — last affects recovery viability), and extension-append logic (original.docx → original.docx.locked).
4
Extract Embedded Public Key
Search binary for PEM header bytes (2D 2D 2D 2D 2D 42 45 47 49 4E = "-----BEGIN") or raw DER-encoded RSA key material. Extract and record — the public key modulus may match keys from prior incidents, establishing same-group attribution (shared key reuse is documented in some groups).
Memory Key Recovery
  • Capture RAM during active encryption — before completion
  • Volatility plugin: aeskeyfind scans memory dump for AES key schedule patterns statistically
  • AES round key expansion has recognizable mathematical properties enabling detection in raw memory
  • Timing constraint: keys exist in memory only during active encryption — acquisition must precede completion; in large environments this window may be 30+ minutes
Pagefile & Hiberfil Recovery
Windows pagefile.sys and hiberfil.sys may contain memory pages including key material paged out during encryption. Extract with Autopsy or direct hex editor. Run bulk_extractor or custom AES key schedule scanner. These artifacts persist through system reboot — unlike RAM — and may be recoverable days post-infection.
NoMoreRansom / ID Ransomware
  • NoMoreRansom.org: Law enforcement / industry partnership providing free decryptors for 160+ families. Always check before paying ransom or declaring data unrecoverable.
  • ID Ransomware: Submit ransom note or encrypted file sample to identify family — may immediately match a decryptable family.
  • Recent decryptors: Avast, Emsisoft, and Bitdefender release new decryptors as operations are disrupted by law enforcement.
Weak RNG Exploitation
If RE reveals weak PRNG (time-seeded rand()): recover the seed from infection timestamp (NTFS $MFT entry creation time or event log entry). Brute-force the PRNG state space within ±60 seconds of identified timestamp. Reconstruct all generated pseudo-random keys. This technique has successfully decrypted multiple early ransomware families where strong RNG was not used.
Code Similarity Analysis
  • BinDiff: Binary-level function comparison between samples — identifies shared code between ransomware families indicating common developer or code reuse
  • TLSH / ssdeep: Fuzzy hashing identifies samples from the same family despite superficial modifications
  • YARA family rules: Match against YARA-Forge community rule sets and VirusTotal YARA retrohunt
Mutex & Artifact Signatures
  • Mutex name: unique per family or campaign — prevents double-encryption and serves as a family fingerprint
  • Language artifacts in ransom notes and error strings reveal operator origin
  • PE compilation timestamp (note: sophisticated actors spoof this — treat as supporting not definitive evidence)
  • PDB path strings in older/debug builds: may reveal developer directory structure or project name
Lazarus Group Ransomware Markers
  • VHD Ransomware, Maui Ransomware — documented DPRK attribution by US CISA/FBI/NSA
  • Code overlap with DPRK wiper malware (WhiskeyAlfa, BLINDINGCAN loaders)
  • Shared infrastructure with Lazarus Group backdoors (COPPERHEDGE, TYPEFRAME)
  • Payment behavior: Lazarus accepts conversion losses — atypical vs profit-motivated criminal groups; suggests state rather than personal financial motivation

Probe the panel with Tor Browser to capture HTTP headers, JavaScript artifacts, and SSL certificate details. Certificate Transparency logs may reveal clearnet infrastructure used during development.

  • Hooks browser rendering / network functions to intercept POST requests before TLS encryption
  • Injects JavaScript into loaded exchange/banking pages — adds fake 2FA fields, records form submissions
  • Man-in-the-Browser (MitB): modifies outgoing withdrawal transactions after victim approves — changes recipient address to attacker wallet
  • Analysis focus: find web injection config (target URL list + JS payload), decrypt it, extract targeted exchange domains
Browser Hook Detection
  • Memory analysis: run malfind on chrome.exe / firefox.exe PID — injected DLLs will appear as MZ-headed memory regions
  • IAT hook detection: compare Import Address Table of browser process against known-clean baseline — hooked functions redirect to injected code
  • Hollowing detection: compare PEB ImageBaseAddress against actual on-disk binary — mismatches indicate process hollowing
Crypto Module Analysis
  • Banking trojans load crypto-targeting capability as a downloaded plugin DLL — extract plugin DLL from C2 download or dropper archive
  • Enumerate: targeted wallet file paths, targeted exchange URLs for form injection, exfiltration endpoint (may differ from main C2)
  • Document all targeted exchange domains — each is a potential victim notification candidate for law enforcement
Major Families
  • TrickBot: Modular; crypto module "pwgrab" specifically targets exchange sessions
  • IcedID / BokBot: Web injection targeting crypto exchanges; often initial access broker for ransomware groups
  • Dridex: AtomBombing injection technique; targets browser sessions for crypto exchanges and web3 applications
  • Qakbot (Qbot): Email thread hijacking delivery; crypto module added post-2022
Mining Config Extraction
  • Extract XMRig (or similar) binary from dropper — usually legitimately-signed XMRig with modified config
  • Mining config embedded as JSON in dropper or downloaded from C2: extract pool URL, wallet address, worker ID, TLS fingerprint, donation level
  • The miner wallet address is the key forensic artifact — trace on Monero explorer and subpoena the mining pool operator for account subscriber records
Process Masquerading Analysis
  • Cryptojackers rename to system process names: svchost.exe with extra space, lsass.exe with capital L
  • Identify via parent PID mismatch: legitimate svchost.exe spawns from services.exe — user-spawned svchost is suspicious
  • Path analysis: legitimate system binaries in C:\Windows\System32; cryptojacker in %APPDATA%, %TEMP%, or user-writable path
  • Volatility: compare pslist vs psscan — rootkit-hidden miners appear in psscan but not pslist
Cloud Cryptojacker Specifics
  • Extract malicious cloud function / container image layer for static analysis
  • Identify IAM role or access key used: trace in CloudTrail who created the compromised credential and when
  • Document unauthorized instance types (p3.*, g4dn.* for GPU mining; c5.* for CPU/XMR) — evidence of criminal gain
  • Estimate victim financial damage: mining revenue (pool payout history) vs cloud compute cost incurred
Persistence Deep-Dive
  • Windows: Scheduled task XML in C:\Windows\System32\Tasks\, Service registration in HKLM\SYSTEM\CurrentControlSet\Services, WMI EventFilter/Consumer subscription
  • Linux: Cron job in /etc/cron.d/ or /var/spool/cron/, Systemd unit in /etc/systemd/system/, LD_PRELOAD rootkit in /etc/ld.so.preload
  • Container: Entry point modification in Docker image layer; Kubernetes DaemonSet creating miner on every node

Smart contract exploit code is the on-chain equivalent of malware — deployed to a blockchain to automatically drain protocol funds. Every step is on-chain and permanently auditable, making this the most transparent attack surface in malware analysis.

EVM Bytecode Decompilation
  • Heimdall-rs: Fast EVM bytecode decompiler → Solidity pseudocode; handles unverified contracts
  • Dedaub Decompiler: Online tool — paste unverified contract address for instant decompilation
  • Etherscan "Decompile Bytecode": Built-in basic decompiler for quick initial analysis
  • If source is verified on Etherscan: read the verified Solidity directly — full source available
Exploit Contract Analysis
  • Identify flash loan integration: AAVE IFlashLoanReceiver, Uniswap V2/V3 swap callback — establishes exploit category
  • Find target protocol address — which contract is being drained and via which function
  • Trace profit extraction: token drained, destination address, flash loan repayment mechanism
  • Document the specific vulnerability: re-entrancy, oracle manipulation, access control, integer overflow — required for legal proceedings and protocol post-mortem
Deployer Wallet Investigation
  • Contract deployer address is on-chain and permanent — trace backward to funding source (exchange withdrawal, bridge, prior address)
  • Pre-attack test transactions: most attackers send tiny test transactions to verify logic minutes/hours before the real exploit — these are in the mempool history
  • Gas price anomalies: exploit transactions use unusual gas pricing for priority block inclusion — identifiable in block history
Vulnerable Code Patterns
  • Re-entrancy: External call before state update — classic DAO pattern. External call (call/transfer) before balance deduction.
  • Oracle manipulation: Single-block price read from AMM pool that was just manipulated by flash loan
  • Missing access control: initialize() callable by anyone; admin functions missing onlyOwner modifier
  • Integer overflow (Solidity <0.8.0): Unchecked arithmetic enabling balance underflow to wrap around to maximum value
